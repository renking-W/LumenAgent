# 会话摘要开发指南

## 一、整体思路

采用**滑动窗口摘要**机制，控制 LLM 上下文长度的同时保留会话连续性。核心思想：不再每次请求加载全量历史消息，而是用 `summary`（已压缩的旧历史）+ `count`（最近未摘要的消息数）代替。

---

## 二、数据模型变更

### sessions 表（SQLite）

在现有 `sessions` 表上新增两个字段：

```sql
-- 现有字段
id         TEXT PRIMARY KEY
created_at TEXT NOT NULL
updated_at TEXT NOT NULL

-- 新增字段
count      INTEGER NOT NULL DEFAULT 0   -- 当前轮次计数器（未摘要的消息轮次数）
summary    TEXT    NOT NULL DEFAULT ''   -- 已摘要的历史（纯文本，由 LLM 生成）
```

**字段说明：**

| 字段 | 含义 | 示例 |
|------|------|------|
| `count` | 自上次摘要以来累积的**未摘要轮次**（每轮 = 1 条 user + 1 条 assistant） | `5` |
| `summary` | 已摘要的旧消息压缩文本 | `用户讨论了项目架构，决定采用 SQLite 存储。` |

### 变更文件

`lumen_agent/infrastructure/sqlite_conversation.py` — `_prepare()` 方法中的 DDL 增加两个字段，并添加数据迁移逻辑（ALTER TABLE ADD COLUMN IF NOT EXISTS 风格）。

## 三、摘要触发逻辑

### 触发条件

每次 `append_message()` 后，`count` 递增。**当 `count >= 10` 时，触发摘要生成。**

### 摘要窗口算法

```
第1次：turns  1-10  →  摘要 turns  1-7，保留 turns  8-10  →  count = 3
第2次：turns  8-17  →  摘要 turns  8-14，保留 turns 15-17  →  count = 3
第3次：turns 15-24  →  摘要 turns 15-21，保留 turns 22-24  →  count = 3
```

每轮窗口 = 7 条摘要 + 3 条保留 = 10 条。保留的 3 条作为下一轮的上下文头部，保证 LLM 感知连续。

### 摘要内容

`summary` 是**累积式**的。每次生成新摘要时：

```
新 summary = 旧 summary + "\n" + 本轮 7 条消息的摘要
```

这样 `summary` 始终包含从会话开始到当前窗口之前的所有关键信息。

### 调用方式

摘要生成调用 LLM 的**非流式 `chat()`** 接口（`DeepSeekHttpClient.chat()`），不使用流式。单次调用，不需要 SSE。

### Prompt 设计

prompt文件会存储在：
```
你是一个严格遵守规则的专业对话增量摘要助手。你的唯一任务是：
基于【上一轮全局摘要】和【本次需要压缩的7轮对话】，生成一段**准确、简洁、无冗余**的新全局摘要。

⚠️ 重要规则：本次只处理【本次7轮对话】，【最近3轮对话】会保留原始内容，不需要你处理。

---

### 核心原则（必须100%遵守，违反任何一条都视为错误）
1. **必须保留的核心信息**（一个都不能丢）：
   ✅ 用户身份、职业、年龄、偏好、禁忌
   ✅ 所有数字、时间、地点、人名、联系方式、订单号
   ✅ 用户的核心需求、已确认的决策、重要约定
   ✅ 待办事项、未完成的任务、下一步计划
   ✅ 工具调用的**最终结果**（绝对不要写调用过程）
   ✅ 用户明确提出的特殊要求和限制

2. **必须彻底删除的内容**（一个都不能留）：
   ❌ 所有闲聊、问候语、告别语、礼貌用语（如"你好"、"谢谢"、"好的"）
   ❌ 重复内容、无意义的确认和附和
   ❌ 模型的自我表述、思考过程、道歉、解释性内容
   ❌ 未确认的假设、推测、主观判断
   ❌ 任何你自己添加的、对话中没有出现的信息
   ❌ 工具调用的中间步骤、参数、错误信息（只保留最终成功结果）

3. **增量更新规则**：
   - 完整继承【上一轮全局摘要】中的所有有效信息
   - 用【本次7轮对话】中的**最新信息覆盖旧信息**
   - 合并重复信息，删除过时、矛盾的信息
   - 不要重新组织上一轮摘要的结构，只做增量补充和更新
   - 不要涉及【最近3轮对话】的任何内容

4. **输出要求（严格执行）**：
   - 摘要长度严格控制在 **150-350字** 之间
   - 用第三人称客观描述，禁止使用第一/第二人称
   - 不要使用任何Markdown格式、表情、特殊字符
   - 直接输出摘要内容，不要加任何前缀、后缀或解释
   - 语言简洁通顺，没有语法错误

---

### 特殊情况处理（覆盖所有边界）
1. 如果【上一轮全局摘要】为空（第一次生成摘要）：
   直接摘要【本次7轮对话】的核心内容
2. 如果【本次7轮对话】为空：
   直接返回【上一轮全局摘要】
3. 如果【本次7轮对话】没有任何有效信息：
   直接返回【上一轮全局摘要】
4. 如果新旧信息有冲突：
   以【本次7轮对话】中的最新信息为准
5. 如果摘要内容超过350字：
   优先保留数字、时间、核心需求，删除次要描述

---
【上一轮全局摘要】：
{{old_summary}}

【本次需要压缩的7轮对话】：
{{seven_rounds_conversations}}

【新全局摘要】：
{{new_summary}}
```

在代码中，prompt的占位符参数（{{}}），需要在调用时自动传入（为空则传"")"。

## 四、消息加载逻辑变更

### 当前行为

```python
# 当前：加载全部历史消息，再截取最近 N 条
history = await repo.list_messages(session_id)
messages = _truncate_context(history, max_context_messages)
```

### 修改后行为

```python
# 修改后：加载 summary + 最近 count 条消息
session = await repo.get_session(session_id)
summary = session["summary"]
count = session["count"]

# 从 messages 表取最近 count 条
recent_messages = await repo.list_recent_messages(session_id, count)

# 拼接最终 LLM 上下文
messages = _build_llm_messages(summary, recent_messages, user_message)
```

其中 `_build_llm_messages` 的逻辑：

```python
def _build_llm_messages(summary: str, recent: list, user_message: str) -> list:
    msgs = []
    if summary:
        msgs.append({"role": "system", "content": f"会话摘要：\n{summary}"})
    msgs.extend(recent)          # 最近 count 条原始消息
    msgs.append({"role": "user", "content": user_message})
    return msgs
```

## 五、API 变更

### ChatRequest：session_id 改为可选

```python
class ChatRequest(BaseModel):
    message: str = Field(..., min_length=1)
    session_id: str | None = Field(None, min_length=1)  # 不传则自动生成新 session_id
```

- 当 `session_id` 为 `None` 或 `""` 时，服务端自动生成一个新的 UUID 作为 `session_id`，实现"不传则开启新会话"
- 传了 `session_id` 则按上述逻辑加载 `summary` + 最近 `count` 条消息

### 新增接口：`GET /v1/sessions/{session_id}/summary`

返回当前会话的摘要信息：

```json
{
  "session_id": "xxx",
  "summary": "用户讨论了项目架构...",
  "count": 3
}
```

## 六、完整数据流（以 10 轮阈值为例）

```
用户发消息 POST /v1/chat { session_id: "abc", message: "你好" }
    │
    ├─ 1. ensure_session("abc")
    ├─ 2. append_message("abc", "user", "你好")
    │       └─ count += 1（假设 count 从 0 → 1）
    ├─ 3. get_session("abc") → { summary: "...", count: 1 }
    ├─ 4. list_recent_messages("abc", count=1) → 最近 1 条
    ├─ 5. 拼接 LLM 上下文：[{system: summary}, {user: "你好"}]
    ├─ 6. llm.chat(messages) → "你好！有什么可以帮你的？"
    ├─ 7. append_message("abc", "assistant", "你好！...")
    │       └─ count += 1（1 → 2）
    │
    └─ ...（重复 1-7，直到 count 达到 10）

当 count 达到 10 时（在第 10 轮 assistant 消息持久化之后）：
    │
    ├─ 8. 触发摘要
    ├─ 9. 加载最近 10 条消息
    ├─ 10. 取前 7 条 + 旧的 summary → 调 LLM chat() 生成新摘要
    ├─ 11. 更新 sessions 表：summary = 新摘要, count = 3
    │
    └─ 后续请求继续走 1-7
```

## 八、边界情况

| 场景 | 处理方式 |
|------|----------|
| count 从未达到阈值 | summary 为空，只传最近 count 条原始消息 |
| 用户第一次发消息且不传 session_id | 服务端自动生成 UUID，count=0, summary="" |
| 摘要生成失败（LLM 报错） | 打印日志，不阻塞用户响应，count 保持不变 |
| 并发请求同一 session | SQLite 写操作通过 `aiosqlite` 的事务锁保证原子性 |
| 重启后加载 | session 表仍在，count 和 summary 直接从数据库读取 |
