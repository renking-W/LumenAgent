## 1. 设计目标

| 目标 | 说明 |
|------|------|
| **模型无关的内部格式** | Agent 核心和消息存储只认一种格式，不依赖任何特定模型 |
| **适配器隔离** | 每个模型一个适配器，负责 内部格式 ↔ 模型格式 互转，互不干扰 |
| **渐进式兼容** | 现有代码和存储数据不改，新格式向后兼容旧数据 |
| **少维护一套转换** | CowAgent 在 Claude 格式上再做一层 OpenAI 转换，是多余的。LumenAgent 应将"统一内部格式"与"各模型格式"严格分层 |

## 2. 核心概念：统一内部格式

### 2.1 为什么需要统一内部格式

前有不同类型的大模型 API 格式：

| 格式 | 代表模型 | tool 消息表示 |
|------|----------|-------------|
| Content Blocks | Claude | `{"type": "tool_use"}` / `{"type": "tool_result"}` |
| Tool Calls | OpenAI / DeepSeek | `tool_calls` / `role: "tool"` |
| Function Call | Gemini | `functionCall` / `functionResponse` |

如果 Agent 核心直接依赖其中任何一种，加新模型就要改核心逻辑。统一内部格式是所有模型格式的"最小公倍数"，让 Agent 核心与模型解耦。

### 2.2 内部消息格式定义

每条消息包含 `role` 和 `content`。`content` 是**内容块列表**，顺序表达语义：

```python
# 用户消息（纯文本）
{
    "role": "user",
    "content": [{"type": "text", "text": "帮我读取文件 config.py"}]
}

# 用户消息（含工具结果）
{
    "role": "user",
    "content": [
        {
            "type": "tool_result",
            "tool_use_id": "call_abc123",
            "content": "文件内容...",
            "is_error": false
        },
        {"type": "text", "text": "基于以上结果继续"}
    ]
}

# 助手消息（思考 + 文本 + 工具调用）
{
    "role": "assistant",
    "content": [
        {"type": "thinking", "thinking": "用户想读取配置文件..."},
        {"type": "text", "text": "我来读取这个文件"},
        {"type": "tool_use", "id": "call_abc123", "name": "read", "input": {"path": "config.py"}}
    ]
}
```

### 2.3 内容块类型一览

| 类型 | 出现位置 | 字段 | 说明 |
|------|----------|------|------|
| `text` | user / assistant | `text: str` | 普通文本 |
| `tool_use` | assistant | `id: str`, `name: str`, `input: dict` | 模型决定调用工具 |
| `tool_result` | user | `tool_use_id: str`, `content: str`, `is_error: bool` | 工具执行结果 |
| `thinking` | assistant | `thinking: str` | 模型推理过程（思维链） |

### 2.4 工具定义格式（模型无关）

```python
{
    "name": "read",
    "description": "Read file contents from workspace",
    "input_schema": {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "File path relative to workspace"}
        },
        "required": ["path"]
    }
}
```

此格式与 Claude API 原生一致，但与 OpenAI/DeepSeek 的 `{type: "function", function: {name, description, parameters}}` 不同。转换由适配器完成。

## 4. 阶段一：统一消息格式与存储升级

### 4.1 当前问题

目前 `messages` 表 `content` 字段存纯文本字符串：

```python
{"role": "user", "content": "帮我读取文件"}
{"role": "assistant", "content": "好的..."}
```

工具场景需要存储混合内容块，纯文本不够用。

### 4.2 改造方案

`messages.content` 从纯文本变为 JSON 序列化的内容块列表：

**存储时：**

```python
import json

# 新消息 → 序列化
content_blocks = [{"type": "text", "text": user_message}]
content_json = json.dumps(content_blocks, ensure_ascii=False)
await repo.append_message(session_id, "user", content_json)
```

**读取时——向后兼容：**

```python
def _parse_content(content: str) -> list[dict]:
    """兼容旧数据（纯文本）和新数据（JSON 数组）。"""
    if not content:
        return []
    try:
        parsed = json.loads(content)
        if isinstance(parsed, list):
            return parsed  # 新格式
    except json.JSONDecodeError:
        pass
    # 旧格式：纯文本 → 包装为 text 块
    return [{"type": "text", "text": content}]
```

### 4.3 `ConversationRepositoryPort` 变更

```python
async def append_message(
    self,
    session_id: str,
    role: str,
    content: str,  # ← JSON 序列化后的内容块列表
) -> None:
    """追加消息。content 始终是 JSON 序列化的内容块列表。"""
    ...
```

`list_messages()` / `list_recent_messages()` 返回的 `content` 字段也是 JSON 字符串，由调用方通过 `_parse_content` 反序列化。

### 4.4 `ChatRequest` / `ChatResponse` DTO

```python
# ChatRequest.message 保持不变，前端只发字符串
class ChatRequest(BaseModel):
    message: str = Field(..., min_length=1)
    session_id: str | None = None

# 历史消息查询 API 返回时标记内容类型
class StoredMessageBlock(BaseModel):
    type: str = "text"         # "text" | "tool_result" | "tool_use" | "thinking"

class StoredMessage(BaseModel):
    role: str
    blocks: list[StoredMessageBlock]  # ← 结构化返回，前端按块渲染
```

### 4.5 应用中如何包装

前端传来的 `message: str` 在 `chat_service.py` 中包装为内容块：

```python
# 当前代码
await repo.append_message(session_id, "user", user_message)

# 改为
content_blocks = json.dumps([{"type": "text", "text": user_message}], ensure_ascii=False)
await repo.append_message(session_id, "user", content_blocks)
```

历史消息加载后通过 `_parse_content` 得到内容块列表，供 Agent 循环使用。

## 5. 阶段二：消息校验与修复

### 5.1 模块定位

```python
# agent/message_unifier.py
```

一个独立模块，不依赖任何模型。在消息进入 Agent 循环前和从 DB 加载后调用。

### 5.2 校验规则

```python
def ensure_message_integrity(messages: list[dict]) -> list[dict]:
    """
    校验并修复消息列表的 tool_use / tool_result 配对完整性。

    规则：
    1. 每个 tool_use 必须有对应的 tool_result（在后续 user 消息中）
    2. 每个 tool_result 必须有前置的 tool_use
    3. 不能以 tool_use 结尾（自动补合成错误结果）
    4. 不能以孤立的 tool_result 开头（应移除）
    5. 移除孤立的 tool_use/tool_result 块

    返回修复后的消息列表（不修改入参）。
    """
```

### 5.3 什么时候调用

| 时机 | 说明 |
|------|------|
| DB 加载历史消息后 | 裁剪/溢出恢复可能破坏配对 |
| Agent 循环每轮开始前 | 工具执行过程可能意外中断 |
| 适配器转换前 | 确保入参干净 |

### 5.4 为什么需要这个模块

不同模型的约束不同：

| 模型 | 约束 |
|------|------|
| Claude | tool_use 后必须紧跟同一用户消息中的 tool_result |
| DeepSeek | 没有紧邻约束，但消息结构错误也会报错 |
| OpenAI | 没有紧邻约束，但 tool_call_id 必须存在且匹配 |

但所有模型共有的底线是：**tool_use 和 tool_result 不能残缺**。将这个最底线的校验放在模型无关层，各模型适配器再加自己的特殊校验。

## 6. 阶段三：模型适配器抽象

### 6.1 `ModelAdapter` 接口

```python
# model_adapters/base.py

from abc import ABC, abstractmethod
from collections.abc import AsyncIterator


class ModelAdapter(ABC):
    """模型适配器：内部格式 ↔ 模型格式 的双向转换。"""

    @property
    @abstractmethod
    def model_name(self) -> str:
        """模型标识。"""
        ...

    async def chat_stream(
        self,
        messages: list[dict],
        tools: list[dict] | None = None,
        *,
        temperature: float | None = None,
        tool_choice: str | dict | None = None,
        max_tokens: int | None = None,
        **kwargs,
    ) -> AsyncIterator[dict]:
        """
        流式调用模型，yield 内部格式块。

        yield 类型：
          {"type": "text", "text": "..."}
          {"type": "thinking", "thinking": "..."}
          {"type": "tool_use", "id": "...", "name": "...", "input": {...}}
          {"type": "stop_reason", "reason": "end_turn" | "tool_use"}
          {"type": "error", "message": "...", "status_code": ...}
        """
        # 标准流程:
        # 1. self.validate_messages(messages)
        # 2. provider_msgs = self._to_provider_messages(messages)
        # 3. provider_tools = self.get_request_parameters(tools)
        # 4. 调用 HTTP 客户端获取 stream
        # 5. for chunk in stream: yield self._to_internal_chunk(chunk)
        ...

    def validate_messages(self, messages: list[dict]) -> list[dict]:
        """
        模型特定的消息校验。
        先调用 ensure_message_integrity() 做通用校验，
        再追加模型特有针对校验证。
        """
        messages = ensure_message_integrity(messages)
        return self._model_specific_validation(messages)

    @abstractmethod
    def _model_specific_validation(self, messages: list[dict]) -> list[dict]:
        """
        模型特定的额外校验。
        DeepSeek：补 reasoning_content 空串
        Claude：  紧邻约束 + 无 role:"tool"
        """
        ...

    @abstractmethod
    def _to_provider_messages(self, messages: list[dict]) -> list:
        """内部格式 → 模型 API 接受的 messages 格式。"""
        ...

    @abstractmethod
    def _to_internal_chunk(self, provider_chunk: dict) -> dict | None:
        """模型 API 返回的一个 chunk → 内部格式块。"""
        ...

    @abstractmethod
    def get_request_parameters(self, tools: list[dict] | None) -> dict:
        """统一内部工具定义 → 模型 API 的工具参数。"""
        ...
```

### 6.2 抽象方法 vs 模板方法

`chat_stream` 已经在基类中提供了模板实现（调用 validate → 转换 → HTTP → 回转），子类只需实现三个抽象方法：

| 方法 | 职责 | 行数预估 |
|------|------|----------|
| `_to_provider_messages()` | 消息格式转换 | ~50-100 行 |
| `_to_internal_chunk()` | 响应块转换 | ~30-80 行 |
| `_model_specific_validation()` | 校验修补 | ~20-50 行 |
| `get_request_parameters()` | 工具定义转换 | ~10-30 行 |

## 7. 阶段四：DeepSeek 适配器实现

### 7.1 类结构

```python
# model_adapters/deepseek.py

class DeepSeekAdapter(ModelAdapter):

    def __init__(self, settings):
        self._client = DeepSeekHttpClient(settings)
        self._tc_buffer: dict[int, dict] = {}  # tool_calls 拼接 buffer

    @property
    def model_name(self) -> str:
        return "deepseek-chat"
```

### 7.2 消息转换：内部格式 → OpenAI 格式

```python
def _to_provider_messages(self, messages: list[dict]) -> list[dict]:
    converted = []
    for msg in messages:
        role = msg.get("role")
        blocks = msg.get("content", [])

        if role == "user":
            has_tool_result = any(b.get("type") == "tool_result" for b in blocks)
            if has_tool_result:
                # tool_result 块 → role: "tool"
                for block in blocks:
                    if block.get("type") == "tool_result":
                        converted.append({
                            "role": "tool",
                            "tool_call_id": block.get("tool_use_id"),
                            "content": block.get("content", ""),
                        })
                # 剩余 text → role: "user"
                text_parts = [b.get("text", "") for b in blocks if b.get("type") == "text"]
                if text_parts:
                    converted.append({"role": "user", "content": "\n".join(text_parts)})
            else:
                text = "".join(b.get("text", "") for b in blocks if b.get("type") == "text")
                converted.append({"role": "user", "content": text or ""})

        elif role == "assistant":
            openai_msg = {"role": "assistant"}
            text_parts, tool_calls, reasoning_parts = [], [], []
            for block in blocks:
                t = block.get("type")
                if t == "text":
                    text_parts.append(block.get("text", ""))
                elif t == "tool_use":
                    tool_calls.append({
                        "id": block.get("id"),
                        "type": "function",
                        "function": {
                            "name": block.get("name"),
                            "arguments": json.dumps(block.get("input", {}), ensure_ascii=False),
                        },
                    })
                elif t == "thinking":
                    reasoning_parts.append(block.get("thinking", ""))
            if text_parts:
                openai_msg["content"] = "\n".join(text_parts)
            if tool_calls:
                openai_msg["tool_calls"] = tool_calls
                openai_msg["content"] = None
            if reasoning_parts:
                openai_msg["reasoning_content"] = "\n".join(reasoning_parts)
            converted.append(openai_msg)

        else:
            converted.append(msg)

    return converted
```

### 7.3 `reasoning_content` 往返

```python
def _model_specific_validation(self, messages: list[dict]) -> list[dict]:
    """
    DeepSeek 特殊要求：
    一旦历史中有 tool_use，所有 assistant 消息必须带 reasoning_content 字段。
    在转成 OpenAI 格式后补空字符串。
    """
    # 先转成 OpenAI 格式
    openai_msgs = self._to_provider_messages(messages)

    # 检查历史中是否有 tool_calls
    has_tool = any(
        msg.get("role") == "assistant" and msg.get("tool_calls")
        for msg in openai_msgs
    )
    if not has_tool:
        return openai_msgs

    # 补 reasoning_content 空串
    for msg in openai_msgs:
        if msg.get("role") == "assistant" and "reasoning_content" not in msg:
            msg["reasoning_content"] = ""
    return openai_msgs
```

> **注意**：这里的 validate 和 convert 边界与 CowAgent 不同。CowAgent 把验证逻辑嵌在 `_convert_messages_to_openai_format` 内部。这里先 validate（并产出 OpenAI 格式消息列表），再发送。职责更清晰。

### 7.4 响应转换：OpenAI 格式 → 内部格式块

```python
def _to_internal_chunk(self, chunk: dict) -> dict | None:
    """DeepSeek stream chunk → 内部格式块。"""
    choices = chunk.get("choices", [])
    if not choices:
        return None
    choice = choices[0]
    delta = choice.get("delta", {})
    finish_reason = choice.get("finish_reason")

    # reasoning_content → thinking 块
    if delta.get("reasoning_content"):
        return {"type": "thinking", "thinking": delta["reasoning_content"]}

    # content → text 块
    if delta.get("content"):
        return {"type": "text", "text": delta["content"]}

    # tool_calls → 拼接 buffer，收齐后转为 tool_use 块
    if "tool_calls" in delta:
        return self._buffer_tool_call(delta["tool_calls"])

    # finish_reason → stop_reason
    if finish_reason:
        # tool_calls buffer 中的工具收齐，一起 yield
        tool_use_blocks = self._flush_tool_buffer()
        return {
            "type": "stop_reason",
            "reason": "tool_use" if finish_reason == "tool_calls" else "end_turn",
            "tool_uses": tool_use_blocks,  # 附加的：本轮所有完整 tool_use 块
        }

    return None
```

### 7.5 流式 `tool_calls` 拼接

```python
def _buffer_tool_call(self, tool_calls_delta: list) -> None:
    """增量拼接 tool_calls。"""
    for tc in tool_calls_delta:
        idx = tc.get("index", 0)
        if idx not in self._tc_buffer:
            self._tc_buffer[idx] = {"id": "", "name": "", "arguments": ""}
        buf = self._tc_buffer[idx]
        if tc.get("id"):
            buf["id"] = tc["id"]
        if tc.get("function", {}).get("name"):
            buf["name"] = tc["function"]["name"]
        if tc.get("function", {}).get("arguments"):
            buf["arguments"] += tc["function"]["arguments"]
    return None  # 增量阶段不 yield

def _flush_tool_buffer(self) -> list[dict]:
    """finish_reason 到达时将 buffer 转为 tool_use 块。"""
    blocks = []
    for idx in sorted(self._tc_buffer.keys()):
        buf = self._tc_buffer[idx]
        try:
            args = json.loads(buf["arguments"]) if buf["arguments"] else {}
        except json.JSONDecodeError:
            args = {}
        blocks.append({
            "type": "tool_use",
            "id": buf["id"],
            "name": buf["name"],
            "input": args,
        })
    self._tc_buffer.clear()
    return blocks
```

### 7.6 `get_request_parameters`

```python
def get_request_parameters(self, tools: list[dict] | None) -> dict:
    """内部格式 → OpenAI tools 格式。"""
    if not tools:
        return {}
    return {
        "tools": [
            {
                "type": "function",
                "function": {
                    "name": t.get("name"),
                    "description": t.get("description"),
                    "parameters": t.get("input_schema", {}),
                },
            }
            for t in tools
        ],
        "tool_choice": "auto",
    }
```

## 9. 阶段六：多模型扩展路径

### 9.1 新增一个模型只需三步

以新增 Claude 为例：

```
第 1 步：model_adapters/claude.py
  class ClaudeAdapter(ModelAdapter):
      _to_provider_messages()     # 内部格式 → Claude API 格式
      _to_internal_chunk()        # Claude chunk → 内部格式块
      _model_specific_validation() # Claude 约束：tool_use/tool_result 紧邻，无 role:"tool"
      get_request_parameters()    # 工具定义 → Claude tools 格式

第 2 步：infrastructure/claude_client.py
  class ClaudeHttpClient:
      """纯 HTTP，不含任何格式逻辑。"""

第 3 步：ModelAdapterFactory.create() 加一条分支
  elif model.startswith("claude"):
      from .claude import ClaudeAdapter
      return ClaudeAdapter(settings)
```

### 9.2 各模型适配器对比

| 模型 | 与内部格式的差异 | 特殊校验 | 预估代码量 |
|------|-----------------|----------|-----------|
| **DeepSeek** | 较大：tool_use→tool_calls，thinking→reasoning_content，tool_result→role:tool | 补 reasoning_content 空串 | ~200 行 |
| **Claude** | 最小：tool_use/tool_result 格式一致，只需处理 role 映射 | tool_use/tool_result 紧邻，无 role:tool | ~120 行 |
| **OpenAI** | 中等：格式与 DeepSeek 近似，无需 reasoning_content | 标准校验 | ~150 行 |
| **Gemini** | 较大：需转 FunctionDeclaration + 特殊角色映射 | 角色约束不同 | ~250 行 |

### 9.3 关于支持多模型的时机

**建议**：初期只实现 DeepSeek 适配器，但**适配器接口一旦定义就不再轻易改动**。接口设计时参考 Claude 和 OpenAI 的特性，避免"DeepSeek 能用但 Claude 不行"的接口问题。
